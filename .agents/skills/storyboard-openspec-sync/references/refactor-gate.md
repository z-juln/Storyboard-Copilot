# 大改重构门禁

功能改动与结构改动**一起评估**。能小步交付时小步；**不能**用「先堆在一个文件里以后再说」代替判断。

## 决策树

```text
本次改动预计影响多少文件 / 多少职责？
│
├─ 小：单职责文件内局部修改（< ~80 行，同一职责）
│     → 直接改，不强制 refactor
│
├─ 中：新能力但可放进已有 application/service 模块
│     → 新逻辑进 service/application；UI/Store 只接线
│
└─ 大：满足任一「必须重构」条件
      → 先拆模块或 refactor，再实现功能（或分两个 commit 说明）
```

## 必须重构（先结构后功能）

满足 **任一** 即应先 refactor 或并行拆文件：

1. **枢纽文件**将新增 **> ~50 行**非 UI 业务逻辑
2. 同一文件已含 **2+ 条**独立业务链（如：树 CRUD + 画布 drop + 预览加载）
3. 新能力与现有代码**命名空间/层级**冲突（如在 Store 写 HTTP 编排）
4. 复制粘贴同一段 manifest/path 逻辑 **≥ 2 处**
5. 用户明确要求「大改 / 重构 / 架构调整」
6. 新子系统没有目录，只能塞进最近的大文件

## 建议重构（评估后向用户说明）

- 目标文件 **> ~400 行**
- 测试/验证需要 mock 整个 React 组件才能测业务
- 新 API 与旧函数堆在同一 **500+ 行** Rust 模块

说明格式：**现状问题 → 建议拆分方案 → 对 diff 范围的影响**；用户要最小 diff 时，至少把**新逻辑**放在新文件，旧文件仅加一行调用。

## 可不 refactor

- 纯样式、文案、常量
- 单职责文件内 bugfix
- 与现有 `application/` 函数同层的 **< ~40 行**扩展
- 紧急 hotfix（但交付中注明 tech debt 与建议拆点）

## 重构范围控制

- **最小有效拆分**：只拆出阻塞新功能的那条链，不顺手大重写
- **沿数据流拆**：UI / application / domain 边界清晰即可
- **不**为 refactor 而新建过度抽象（单函数 helper 文件、无复用价值的 interface 层）
- refactor 与 feature 同一 PR 时，交付说明写清 **结构变更** 与 **行为变更**

## 与文档同步的关系

若 A 阶段新增目录、`application/` 模块、HTTP 路由或 invariants → B 阶段**必须**更新 openspec（见 [sync-checklist.md](sync-checklist.md)）。
